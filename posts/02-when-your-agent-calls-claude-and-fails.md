# 让 AI agent 去调 Claude 帮你干活——然后她翻车了

> 我让我的 AI agent 调 Claude Code 去搞定一个复杂任务。她很努力，Claude 也很努力，但两个人联手折腾了一整天，愣是没搞定。第二天我亲自下场，换了个问法，几小时就通了。问题不在工具，在于**怎么问**。

## 背景

我的 agent 不是只会聊天的花瓶——她能调 Claude Code CLI 来处理编码任务。简单说就是：你跟她说"帮我搞定 XXX"，她会把任务打包成一个 prompt，甩给 Claude 去写代码、跑调试，然后把结果带回来。

听着很美对吧？大部分时候确实好使。

直到她遇上了一个硬骨头。

## 翻车现场

任务本身不算离谱：把本地文件上传到某个网盘，然后返回一个分享链接。

但这个网盘的上传协议……怎么说呢，它不是那种规规矩矩的 REST API，而是一套分片上传 + 回调认证 + 多阶段握手的复杂流程。浏览器里点一下"上传"很简单，但背后的 HTTP 链路长得像俄罗斯套娃。

我的 agent 很勤快。她先让 Claude 抓了浏览器的完整上传 trace，拿到了每一步的真实请求。然后开始照着 trace 去复刻。

```
pre_upload → upload/auth → PUT 分片 → complete(回调) → finish
```

看起来方向没问题。

然后就卡住了。

complete 阶段一直返回 203，回调报 400。她让 Claude 调了一遍又一遍——换 header、改 content-md5 的算法、对齐 XML 格式、检查 ETag……

**整整一天，同一个 error，同一个方向。**

## 问题出在哪

不是 Claude 不行，也不是 agent 不努力。问题出在一个看起来不起眼的字段：

这个网盘的上传协议有多条分支。浏览器走的那条恰好是最复杂的那条——它需要实现一个增强型的哈希上下文协议，要额外传递几个特殊的请求头。

但还有一条更基础的分支。只需要在请求时告诉服务端"我不走增强协议"，然后老老实实传 md5 和 sha1 就行了。

**我的 agent 抓到了浏览器的"真相"，但真相只是"浏览器当时选择的那条路"，不等于"唯一的路"。**

她把"必须完全复刻浏览器"当成了唯一正解，然后在最复杂的那条分支上越陷越深。Claude 的执行力很强，给什么方向就往哪冲——但方向错了，执行力越强，离目标越远。

## 我是怎么搞定的

第二天我接手后，没有继续在原来的方向上微调。

我给 Claude 的第一句话不是"修这个 bug"，而是：

> "核心目标是把本地文件上传到网盘，然后返回分享链接。先梳理可能的实现路径，评估哪条最可行，再开始写代码。"

注意区别：**不是让 Claude 继续修，而是让它退一步看全局。**

Claude 重新分析后很快发现了那个协议开关——增强模式需要一套我们没实现的上下文传递机制，这才是 203 错误的根因。关掉它，走基础分支，补上文件哈希，上传就通了。

之后又修了几个工程细节（文件名编码、分享接口的重定向处理），整个流程就跑通了。

## 复盘：agent 和 Claude 的协作哪里掉链子了

梳理完整个过程，我发现问题不在工具本身，而是**给 Claude 的 prompt 策略**有硬伤：

### 1. 一上来就让 Claude 冲细节

agent 的第一个 prompt 就是"把 complete 阶段的 header 对齐，content-md5 要这样算……"

这等于直接把 Claude 锁死在了执行层。它根本没机会思考"这条路对不对"。

**正确做法：第一个 prompt 应该是战略层。** "有几条路？各自什么难度？推荐哪条？先给方案，别写代码。"

### 2. 卡住了还在同一个方向上死磕

同一个 error 反复出现，agent 的反应是：让 Claude 继续调参数、换算法、改格式。

**但有时候 error 本身就是在告诉你：你走错路了。**

第 3 次失败的时候，就应该停下来问：我是不是在错误的分支上？有没有完全不同的方式绕过去？

### 3. 过拟合"浏览器真相"

抓包拿到的是浏览器的行为，不是唯一可行的行为。同一个 API 可能有多条协议路径，浏览器走的那条未必是最适合你的那条。

逆向工程不是"无脑复刻"，而是"理解协议，然后选择最适合自己的路径"。

## 升级方案

翻完这个车之后，我给 agent 加了几条硬规则：

**三振出局：** 同一个方向连续失败 3 次，必须暂停，切换到诊断模式。不准第 4 次、第 5 次继续微调同一个参数。

**分层 prompt：** 复杂任务必须分两步调 Claude。第一步是诊断层——"有几条路？推荐哪条？"第二步才是执行层——"确认走方案 X，开始实现。"

**卡住了重新诊断：** 不是"继续修"，而是"退一步看"。让 Claude 重新评估当前方向是否正确，有没有完全不同的绕行方案。

**及时止损：** 如果同一个 error 上花了超过一定时间，或者 Claude 调用超过 5 次还没进展，必须暂停向我汇报，不许独自死磕。

```
# 诊断层 prompt 模板
"当前目标是 [XXX]。请先分析：
1. 有几种实现路径？
2. 各自的难度和风险？
3. 推荐哪条？为什么？
先给方案，不要写代码。"

# 执行层 prompt 模板
"确认走方案 X。请实现。
先用最小文件跑通完整闭环，
遇到非预期 error 先暂停分析，不要自动 fallback。"

# 卡住时 prompt 模板
"当前卡在 [error]，已尝试 [1/2/3]。请重新诊断：
1. 这个 error 是否说明我们在错误的路径上？
2. 有没有完全不同的方式绕过？"
```

## 这次经历教会我什么

**方向错了，执行力越强死得越快。**

这句话不只适用于 AI agent。写代码也是、做产品也是、创业也是。

当你发现自己在一个问题上反复打转，不断地"再试一次"、"微调一下"、"应该差不多了"——

停下来。

问自己一个问题：**我是不是在错误的路上跑得太用力了？**

有时候解决问题的方式不是跑得更快，而是换一条路。
